1.1 进程的几个特殊定时器

1.1.1 概述

Linux允许用户进程激活特殊的时间定时器----间隔定时器，这些定时器使得可以周期性的发送信号给当前进程。当然也可以设置成在经过一段时间延迟之后只发送一个信号。

每种定时器因此具有如下特征：

1.         定时器时间频率，也就是每隔多长时间发送一次信号，如果该值为0，则表示仅发送一次；

2.         现在距下一次信号产生还剩余多长时间；

Linux中通过setitimer系统调用来设置这几个定时器的参数，该函数的第一个参数用来指定设置何种类型的定时器：

1.         ITIMER_REAL：实时定时器，表示系统一旦实际运行了指定的时间片段之后，立刻发送SIGALRM信号给该进程。该定时器是实时的，不管进程是在运行还是休眠中，都会立刻执行发送信号给该进程；

2.         ITIMER_VIRTUAL：用户空间非实时定时器，该定时器只会计算该进程在用户空间运行的时间片，当进程在用户空间累计运行的时间达到定时器时间间隔时，才会发送SIGVTALRM信号给该进程。如果该进程处于睡眠状态或者系统空间运行状态，则该定时器不会对此时间片进行计时；

3.         ITIMER_PROF：非实时定时器，和前者一样，唯一不同的时，该定时器会计算进程在系统空间运行的时间片。

为了实现上述三种类型的定时器策略，task_struct中包含了如下三对成员域：

Ø  it_real_incr 和 it_real_value

Ø  it_virt_incr和it_virt_value

Ø  it_prof_incr和it_prof_value

每对中的第一个域保存了时间间隔数值，第二个域保存了距离下一次定时器事件的时间值。对于ITIMER_REAL类型的定时器，由于是实时的，即使进程在休眠中，也必须把信号发送给进程。因此其采用timer定时器机制实现，为此task_strcut中包含了一个timer对象tsk->real_timer。他们在do_fork中的初始化如下所示：

1.1.2 ITIMER_REAL类型定时器的实现

对于task_struct->real_timer.function，是在INIT_TASK宏中就已经初始化好了的，为函数it_real_fn，该初始化将会随着fork/clone/kernel_thread/vfork一直传递给所有的子进程。

对于task_struct->real_timer.data和list，则是在do_fork时初始化好的，如下面的do_fork中的620、621行所示；

而task_struct->real_timer.expires以及add_timer的调用则是在系统调用do_setittimer中，我们来看看do_setitimer的实现：

 

int do_setitimer(int which, struct itimerval *value, struct itimerval *ovalue)

{

    register unsigned long i, j;

    int k;

 

    i = tvtojiffies(&value->it_interval);

    j = tvtojiffies(&value->it_value);

    if (ovalue && (k = do_getitimer(which, ovalue)) < 0)

        return k;

    switch (which) {

        case ITIMER_REAL:

            del_timer_sync(&current->real_timer);

            current->it_real_value = j;

            current->it_real_incr = i;

            if (!j)

                break;

            if (j > (unsigned long) LONG_MAX)

                j = LONG_MAX;

            i = j + jiffies;

            current->real_timer.expires = i;

            add_timer(&current->real_timer);

            break;

    ......

 

这里传下来三个参数，which==ITIMER_REAL表示设置ITIMER_REAL定时器，并设置task_struct->real_timer.expires=value->it_interval，同时调用add_timer（&current->real_timer)启动该定时器；注意这里还将value->it_interval保存在task_struct->it_real_incr中，用于定时器处理函数中使用，而对于task_struct->it_real_value实际上一直没有用到；再来看看it_real_fn的实现：

 

void it_real_fn(unsigned long __data)

{

    struct task_struct * p = (struct task_struct *) __data;

    unsigned long interval;

 

    send_sig(SIGALRM, p, 1);

    interval = p->it_real_incr;

    if (interval) {

        if (interval > (unsigned long) LONG_MAX)

            interval = LONG_MAX;

        p->real_timer.expires = jiffies + interval;

        add_timer(&p->real_timer);

    }

}

 

可见，当定时器到期后，立刻给p所指向的进程发送SIGALRM信号，p就是在do_fork中创建的子进程，也是do_setitimer中的current。信号发送完毕之后，如果发现p->it_real_incr，也就是前面do_setitimer中保存的时间间隔，不为0，则启动下一次定时器，时间间隔依然为p->it_real_incr，如果为0，则不再启动定时器，也就是仅仅发送一次信号。

1.1.3 ITIMER_VIRTUAL和ITIMER_PROF类型定时器的实现

对于ITIMER_VIRTUAL和ITIMER_PROF类型定时器的实现并没有采用timer定时器机制。这是为什么呢？因为他们的更新要求进程必须在运行中，没有必要采用timer定时器机制。每次时间中断到来后，都会调用时间中断处理函数timer_interrupt，随后à do_timer_interruptà do_timerà update_process_timesà update_one_process，我们来看看update_one_process函数代码：

 

void update_one_process(struct task_struct *p, unsigned long user,

            unsigned long system, int cpu)

{

    p->per_cpu_utime[cpu] += user;

    p->per_cpu_stime[cpu] += system;

    do_process_times(p, user, system);

    do_it_virt(p, user);

    do_it_prof(p);

}  

 

该函数是每次时钟中断都会调用的，do_it_virt用于处理进程的用户空间非实时定时器，do_it_prof用于处理进程的非实时定时器，我们来详细看看这两个函数的实现过程。

1.1.3.1 do_it_virt

我们在前面说过：用户空间非实时定时器，该定时器只会计算该进程在用户空间运行的时间片，当进程在用户空间累计运行的时间达到定时器时间间隔时，才会发送SIGVTALRM信号给该进程。如果该进程处于睡眠状态或者系统空间运行状态，则该定时器不会对此时间片进行计时；另外，task_struct->it_virt_value保存了距离下一次定时器事件的时间值，task_struct->it_virt_incr保存了时间间隔数值。每次时钟中断发生时，系统都会中断掉正在运行的进程，立刻响应中断并进入中断处理过程。而被中断掉的进程可能正在用户空间运行也可能正在系统空间运行，如果在用户空间中运行，那么中断时CPU保存在堆栈中的CS一定__USER_CS，否则就一定时__KERNEL_CS，这两个定义的值如下：

 

#define __KERNEL_CS 0x10

#define __KERNEL_DS 0x18

#define __USER_CS   0x23

#define __USER_DS   0x2B

 

#define user_mode(regs) ((VM_MASK & (regs)->eflags) || (3 & (regs)->xcs))

 

因此我们在中断处理时，只要判断当前压入到栈中的CS就可以知道中断是发生在当前运行进程运行在用户空间还是系统空间，这里user_mode判断栈中的CS&3是否为真，很明显，对于内核空间而言，CS&3==0，宏结果为0，对于用户空间而且，CS&3==3，宏结果为1（注，对于前一个判断条件(VM_MASK & (regs)->eflags)我们这里暂不解释）。

现在我们来看do_it_virt函数的实现：

 

538 static inline void do_it_virt(struct task_struct * p, unsigned long ticks)

539 {

540     unsigned long it_virt = p->it_virt_value;

541

542     if (it_virt) {

543         it_virt -= ticks;

544         if (!it_virt) {

545             it_virt = p->it_virt_incr;

546             send_sig(SIGVTALRM, p, 1);

547         }

548         p->it_virt_value = it_virt;

549     }

550 }

551

 

对于需要循环发送信号的进程而言，p->it_virt_incr和p->it_virt_value都不为0。不放假设为进程A调用了do_setitimer设置了循环发送信号，如果当前定时器中断了正在运行的进程为进程A，则这里首先获取保存在进程A的p->it_virt_value中距离下一次定时器事件的时间值到it_virt，如542行，对于进程A第一次被时钟中断并进入该函数中，it_virt为进程A调用do_setitimer设置的初始值，必然不为0，而在以后因为it_virt递减的原因==0的时候，则又被下面的544行截取，随即将it_virt设置为进程A的该定时器时间间隔p->it_virt_incr，因此542在此种情况下，永远为真。如果进程A是在用户空间运行时被中断的，则ticks= user_mode(regs)的值为1，因此，给it_virt递减，如果递减后值为0，则表示定时器事件到期，发送SIGVTALRM信号，并且重新设置it_virt为进程A的该定时器时间间隔p->it_virt_incr，表示将要重新开始计数等待下一次定时器时间的到来。如果进程A是在系统空间运行时被中断的，则什么也不做。最后548行将当前的局部变量it_virt保存到p->it_virt_value中，表示距离下一次定时器事件的时间值。

对于仅需要发送一次信号的进程而言，其p->it_virt_value为该信号发生的时刻，而p->it_virt_incr则=0，因此当it_virt递减到0时，引起发送信号，但是it_virt被设置为p->it_virt_incr==0，下一次进入该函数时，由于it_virt==0，因此经过542行时就直接退出函数了。

对于没有调用do_setitimer的进程而言，p->it_virt_incr和p->it_virt_value均为0，故无论何时经过542行时就直接退出函数了。

这里需要注意的是，该定时器的时间片只是针对进程A正在运行时，而且是正运行在用户空间时才统计的，故可以认为是当进程A在用户进程实际运行了p->it_virt_inc时间片之后，才会发送SIGVTALRM信号，故为非实时的。

1.3 进程组和会话

1.3.1 设备文件

在Linux系统中，存在一个抽象化的设备目录，名为/dev。该目录下存有指向系统中硬件的特殊文件。这些指向硬件设备的文件，极大地简化了程序员对硬件的操作。因为，程序员就可以像访问普通文件一样来访问硬件，而无需使用特殊的接口函数。Linux系统将设备分成了3种类型：字符设备、块设备和网络接口。

l  字符设备：字符设备就是能够像字节流一样访问的设备，字符终端和串口就属于字符设备。

l  块设备：块设备上可以容纳文件系统。与字符设备不同，在读写操作时，块设备每次只能传输一个或多个完整的块。在Linux操作系统中，应用程序可以像访问 字符设备一样读写块设备（一次读取或写入任意的字节数据）。因此，块设备和字符设备的区别仅仅是在内核中对于数据的管理不同。

l  网络接口：网络接口存在一定的特殊性，其与内核的通信完全不同于内核与字符设备或块设备的通信。网络接口可以是一个硬件设备，也可以是一个软件设备。

ls Cl位于/dev目录下设备，字符设备的话，将以字符“c”开头。块设备将以字符“b”开头：

crw-------     1 root     root      26, 0   Mar 23  2001 wvisfgrab

brw-rw----    1 root     disk      13, 0   Mar 23  2001 xda

查看/dev下的设备文件名，可以看到，每个设备都 带有诸如“14，4”或“3，0”这样的编号。使用ls命令列出所有以hda开头的设备名称：

brw-rw----    1 root     disk       3, 1   Mar 23  2001 hda1

brw-rw----    1 root     disk       3, 2   Mar 23  2001 hda2

这种号码称为主从设备号，内核通过该号码将设备文件与真实设备关联起来。前面的数字称为主设备号，指向硬件设备，而次设备号指向子设备。由同一个设备驱动控制的所有设备具有相同的主设备号。从设备号则被用来区分具有相同主设备号且由相同的设备驱动控制的不同设备。例如上图中，/dev/hda1和/dev/hda2设备文件的主设备号均为3，次设备号分别为1和2。

在Linux系统启动时，设备驱动将被加载。设备驱动成功加载后，将向系统反馈一个主设备号，驱动程序将根据该主设备号在/dev目录下创建对应的设备文件。这样，程序就可以使用open、read、write函数或命令，通过设备文件实现对设备的访问了。

1.3.2 终端设备

终端为显示器、控制器及键盘等设备的总称。终端与平常使用的计算机的根本区别在于：没有自己的CPU单元和内存单元。终端的主要功能是发送用户输入的请求给主机，并显示主机完成运算后的结果。随着计算机技术的发展，目前终端有了新的含义。一切接入网络的计算机设备称为终端，如电脑、PDA、智能电话、数字电视等。

在Linux系统中，所有终端设备都被称为tty。 tty被认为是Teletypes或Tele-typerwriters的缩写。终端一种字符型的设备，有多种类型。查看/dev/目录下的以tty开头的设备名称，可以了解Linux系统中所有的终端设备。Linux系统中的终端设备文件主要有下面几种类型。

l  串行端口终端（/dev/ttySn）：串行端口终端是使用计算机的串行端口连接的终端设备，在Linux系统中，系统将每个串行接口看作是一个字符设 备。串行端口对应的设备名称是/dev/ttyS0、/dev/ttyS1、/dev/ttyS2、/dev/ttyS3，如图6.6所示分别对应于硬件 的COM1、COM2、COM3和COM4。例如执行命令ls Cl /dev/ttyS*。

l  伪终端（/dev/pts/n）：使用Telnet登录到Linux系统实际上是登录到了一个伪终端上。使用who命令可以看到其位于伪终端上。

l  控制终端（/dev/tty）：如果当前进程存在控制终端，/dev/ttyN就是当前进程的控制终端的设备文件。使用“ps-ax”命令可以查看到进程与控制终端的关系。

l  控制台终端（/dev/ttyn,/dev/console）：在Linux系统中，常将显示器称为控制台终端。当从控制台登录到系统时，使用的是tty1。Linux系统将tty1～tty6称为虚拟终端。

通过stty命令可以直接修改和查询终端驱动程序的设置。有时候运行一些程序，需要修改终端的设置。对于用户而言，这些修改往往是不可见的，因为程序在退出后，会将终端设置为原有的情况。但是，如果出现程序崩溃或被kill命令杀死的情况，有可能来不及 将终端设置为原有的参数。这时，终端就有可能出现异常的情况，有可能不能正常处理换行字符，或不能恰当地显示输入的字符，甚至可能不能正确执行命令。使用cat命令或more命令去显示一个二进制文件也有可能会造成终端的混乱。在这种情况下，可以使用reset命令将终端恢复到正确的状态。Linux系统中提供了ttyname函数，用于获得当前终端名称。使用tcgetattr函数与tcsetattr函数控制终端。

2.         init对每一个tty执行一次fork，fork后的子进程相应的执行/sbin/mingetty ttyX。随后mingetty输出“login:”之类的信息，等待用户输入用户名；

3.         用户输入用户名后，getty就完成了，随后其按如下方式调用login：execle("/usr/bin/login", "login", "-p", username, (char *) 0, envp); getty自己找到配置文件中的环境变量并传递给login，如果login登录失败，则测试的父进程init知道了错误信息，就会再次为此ttyX调用fork，随后又执行/sbin/mingetty ttyX，如此往返。

4.         如果登录正确，login就将当前工作目录更改为该用户的起始目录(chdir)。它也调用chown改变该终端的所有权，使该用户成为所有者和组所有者。将对该终端设备的存取许可权改变成：用户读、写和组写。调用setpgrp设置进程的组ID。然后用login所得到的所有信息初始化环境：起始目录(HOME)、shell(SHELL)、用户名(USER和LOGNAME)，以及一个系统默认路径(PATH)。最后，login进程改变为登录用户的用户ID(setuid)并调用该用户的登录shell，类似于：execl("/bin/sh", "-sh", (char *) 0); 这里的sh也可以为bash等shell，到此为止，登录用户的登录shell开始运行，其父进程是init，所以当该shell终止时，init会得到通知（接到SIGCHLD信号），init会对该终端全部重复上述过程。

5.         登录shell将文件描述符0，1和2设置为终端设备。现在，登录shell读其起动文件(.bashrc，.bash_profile等)。这些起动文件通常改变某些环境变量，加上一些环境变量。例如，很多用户设置他们自己的PATH等。当执行完起动文件后，用户最后得到shell的提示符，并能键入命令。

1.3.5 什么是进程组？

在Linux系统中，所有的进程都属于各自的进程组。进程组是一个或多个进程的集合。每个进程组都有类似于进程号的标识，称为进程组ID。进程组ID是由领头进程的进程号PID决定的，每个进程组都存在一个领头进程。进程组的存在与否与领头进程是否存在没有关系。也就是说当领头进程死掉了之后，这个进程组就没有领头进程了，一旦进程组中所有的进程全部都死掉了或者执行setpgid跑光了，则该进程组也就消失了。子进程的进程组ID将会继承父进程的进程组ID。

进程可以调用setpgid(pid,pgid)来改变自己或者自己的子进程所属的进程组（但如果子进程和当前进程的会话不一样，或者子进程已经执行了exec，则不可设置。另外如果pgid!=pid，而且pgid对应的进程组已经存在，则设置要改变的进程的进程组ID为pgid；如果pgid==pid，则直接以pid为pgid创建新的组）。

应用举例：进程组可以抽象成为“工作”，例如：$ ls | sort | more命令，对于支持进程组的shell例如bash，将会为该三个进程创建一个新的进程组，该进程组的ID为领头进程的PID（应该是对ls、sort和more进程setpgid是他们成为同一个进程组）。

1.3.6 什么是会话？

 

会话是一个或多个进程组的集合。与进程组类似，每个会话都存在一个领头进程。Linux是一个多用户的操作系统，在同一时刻系统中会存在属于不同用户的多个进程。

当用户从终端登录系统（不管是终端还是伪终端）时，例如ssh、telnet客户端程序，或者本地的mingetty（在本地显示器上显示login：的程序），该进程将会调用setsid创建一个会话，该进程也就成为了该会话的领头进程，而该进程打开的终端也就成为该会话的控制终端。在该终端上启动的进程都会被系统划归到会话的进程组中。会话中的进程通过该会话中的领头进程与一个终端相连。该终端是会话的控制终端。一个会话只能有一个控制终端，反之一样。如果会话存在一个控制终端，则它必然拥有一个前台进程组。属于该组的进程可以从控制终端获得输入。这时，其他的进程组都为后台进程组。产生在控制终端上的输入和信号将发送给会话的前台进程组中的所有进程，终端上的连接断开时 (比如网络断开或 Modem 断开)，挂起信号将发送到控制进程(session leader)。如果用户在某个终端上发送了某个信号，例如，按下“Ctrl+C”发送SIGINT信号，那么此时通过终端找到会话领头进程，会话领头进程然后给前台进程组中所有的进程发送SIGINT信号。

会话和进程组是Linux内核用于管理多用户情况下用户进程的方法。每个进程都属于一个进程组，而进程组又属于某个会话。

进程可以调用setsid()建立一个新的会话，前提是该进程不能为某个进程组的leader。创建完毕后，该进程成了新会话的leader，新进程组的leader，没有可控制的tty，其task_struct->session = task_struct->pgrg = task_struct->pid，当前进程成为新会话的唯一进程。

setpgid的实现如下：

 

asmlinkage long sys_setpgid(pid_t pid, pid_t pgid)

{

    struct task_struct * p;

    int err = -EINVAL;

    if (!pid)       pid = current->pid;

    if (!pgid)      pgid = pid;

    if (pgid < 0)   return -EINVAL;

    read_lock(&tasklist_lock);

 

    err = -ESRCH;

    p = find_task_by_pid(pid);

    if (!p)     goto out;

 

    if (p->p_pptr == current || p->p_opptr == current) {

        err = -EPERM;

        if (p->session != current->session)

            goto out;

        err = -EACCES;

        if (p->did_exec)

            goto out;

    } else if (p != current)

        goto out;

    err = -EPERM;

    if (p->leader)      goto out;

    if (pgid != pid) {

        struct task_struct * tmp;

        for_each_task (tmp) {

            if (tmp->pgrp == pgid &&

                tmp->session == current->session)

                goto ok_pgid;

        }

        goto out;

    }

 

ok_pgid:

    p->pgrp = pgid;

    err = 0;

out:

    /* All paths lead to here, thus we are safe. -DaveM */

    read_unlock(&tasklist_lock);

    return err;

}
