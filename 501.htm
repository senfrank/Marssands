1.2 80x86 CPU对进程的支持

1.2.1 进程切换概述

为了控制进程的执行，内核必须能够挂起正在当前CPU中运行的进程并且恢复先前挂起的其他进程。这个行为称之为进程切换。所有的进程都共享CPU寄存器，所以在恢复执行一个进程前，内核必须保证每个这样的寄存器被正确的加载到其以前的状态值。这组将要被加载到寄存器中的值称之为硬件上下文。硬件上下文是进程执行上下文的子集。进程执行上下文包括进程运行所需的各种信息。在Linux中，部分硬件上下文保存在进程的task_struct中，其余的则保存在内核栈中。

1.2.2 任务段TSS及X86 CPU的进程切换

X86 CPU中支持段，例如CS段、DS段、ES段、SS段、FS段等等，一个段就是一段具有特定意义的内存，该段内存的基地址、长度、访问权限等保存在一个8字节的段描述符中，而该段描述符又存放在全局段描述表GDT中。如果操作系统希望访问某个段，那么操作系统首先把该段的段描述符在全局描述符表中的位置及访问权限加载到相应的段寄存器中（例如代码段寄存器CS），CPU依据段寄存器中的索引在GDT中找到相应的段描述表项，经过权限比较之后，依据段描述中保存的段基地址就找到对应的段在内存中的位置了，从而实现了访问该段的目的。这里该段的段描述符在GDT中的位置，是操作系统在初始化该段时决定的，同时操作系统一并初始化了该段描述符。

X86中为了支持进程切换，又另外增加了任务段（Task-State Segment：TSS）。和前面的CS段、DS段等段一样，TSS也有相应的段描述符，也需要在GDT中占用一个表项。不同的是，TSS大小恒定为104字节，每个进程都有必须有一个TSS，其中保存了该进程被挂起时刻各寄存器的值以及其他重要信息，主要用于进程切换功能的支持，TSS中的内容如下所示：

X86 CPU是如何利用TSS来实现进程切换的呢？X86 CPU要求操作系统每创建一个进程时，都必须为其创建一个TSS。操作系统在创建该进程时，首先在内存中为该进程创建一个104字节的TSS并适当初始化，然后在GDT中找到一个合适的位置I，确保该位置当前没有人使用后，把刚刚创建的TSS基地址、段长度、访问权限等写入到该描述符表项中。下面我们举例说明切换过程：当操作系统准备调用jmp B_TSS:[address]等原因导致任务切换时，不妨假设从任务A切换到任务B，此时CPU将当前所有寄存器的值保存在A->TSS中，然后依据B_TSS在GDT中找到B的TSS段描述符，并依次找到B->TSS，找到后将B->TSS中保存的寄存器值恢复到CPU各寄存器中，此刻便导致B进程开始执行，从而实现了任务的切换。这里的B_TSS和CS段寄存器内容类似，是个TSS segment selector，16bit，高13位保存该TSS段描述符在GDT中的索引，低3位保存访问权限设置，和CS、DS等段选择符含义类似。X86 CPU访问某个进程TSS的过程图如下所示：

1.2.3 TSS段寄存器

X86 CPU中还存在TSS段寄存器，对进程的切换用处不大，其相关描述如下：The task register holds the 16-bit segment selector and the entire segment descriptor (32-bit base address, 16-bit segment limit, and descriptor attributes) for the TSS of the current task (see Figure 2-5). This information is copied from the TSS descriptor in the GDT for the current task. Figure 6-5 shows the path the processor uses to access the TSS (using the information in the task register). The task register has a visible part (that can be read and changed by software) and an invisible part (maintained by the processor and is inaccessible by software). The segment selector in the visible portion points to a TSS descriptor in the GDT. The processor uses the invisible portion of the task register to cache the segment descriptor for the TSS. Caching these values in a register makes execution of the task more efficient. The LTR (load task register) and STR (store task register) instructions load and read the visible portion of the task register:

The LTR instruction loads a segment selector (source operand) into the task register that points to a TSS descriptor in the GDT. It then loads the invisible portion of the task register with information from the TSS descriptor. LTR is a privileged instruction that may be executed only when the CPL is 0. It’s used during system initialization to put an initial value in the task register. Afterwards, the contents of the task register are changed implicitly when a task switch occurs.

The STR (store task register) instruction stores the visible portion of the task register in a general-purpose register or memory. This instruction can be executed by code running at any privilege level in order to identify the currently running task. However, it is normally used only by operating system software. On power up or reset of the processor, segment selector and base address are set to the default value of 0; the limit is set to FFFFH.

1.3 Linux的进程切换实现机制

Linux本身的进程切换并不使用X86 CPU提供的切换机制，但是由于X86 CPU要求软件必须设置TR和TSS，因此内核在初始化时就走过场创建了一个TSS（init_tss），并设置TR指向该TSS用来满足CPU的要求。但是设置完毕之后，就永远不再修改了（除了TSS的SS0:ESP0，后面详叙）。内核不会使用X86 CPU的JMP等来实现任务的切换，也不依靠TSS来保存每个进程切换时的寄存器副本，而是将这些副本保存在各个进程自己的系统空间堆栈中，这样一来，TSS中的内容几乎就没有用了，其唯一剩下的作用就是CPU自动更换堆栈会使用到，下面来详细解释。

我们知道，当运行级别发生从低到高的变化时（例如Linux中从用户空间到系统空间），会使得CPU自动更换堆栈（为什么要自动更换，主要是保护模式下，CPU不允许软件从运行级别低的状态访问运行级别高的空间），而CPU更换堆栈时，会从TR对应的TSS中找到更换后的级别对应的SS:ESP，并加载其到SS和ESP寄存器中，同时还会依据IDT等表执行更换后的级别中的处理函数（这点就无需进程来参与了，对于Linux而言，除了中断、异常和系统调用外，没有其他方式从低级向高级转化了，而这几者的处理都属于内核按照CPU要求预定义好的系统功能，其实是进程无关的。只有堆栈和进程相关）。X86 CPU支持4个运行级别。Linux系统只只使用了其中两个运行级别：0（高）和3（低），0表示系统空间，3表示用户空间，从0到3级的转换由Linux操作系统自行完成，也不会涉及到CPU的介入。但是从3级到0级的转换，例如当进程运行在用户空间时，如果此时来了个中断、异常或者系统调用，则此时必然引起CPU自动更换堆栈，并切入到内核运行，那么此时CPU如何获取到SS0和ESP0呢？前面已经说过，内核初始化时会建立一个TSS（init_tss）并让TR永久指向该TSS，同时将init_tss的SS0:ESP0设置为内核初始化栈为init_task_union。当内核实施进程切换时，会将被换入的进程的系统空间堆栈SS:ESP的值保存到init_tss的SS0:ESP0中。当该进程因为中断等原因进入系统空间时，引起CPU自动堆栈切换，CPU从TR->TSS也就是init_tss中取得0运行级别对应的堆栈SS0:ESP0，并加载到SS:ESP寄存器中，使得堆栈从用户空间堆栈切换到该进程对应的系统空间栈。Linux中唯一初始化（单CPU）的TSS段如下图所示：

1.4 关于LDT

在Linux内核中基本上不使用局部段描述符表LDT。LDT只是在VM86模式中运行wine以及其他在Linux上模拟运行Windows软件或DOS软件的程序中才使用。

1.5 进程需要系统堆栈的原因？

假设进程正在用户空间运行，此时突然来了个中断，此时CPU因为中断而从用户空间进入系统空间，并且会由于运行级别的变化而自动更换堆栈。这个过程是如何呢？CPU依据当前进程的TRàGDT(n)àTSS获取该TSS中的SS0以及ESP0，然后将当前的AX,BX,….,CS,DS,…..等等保存到SS0:ESP0的堆栈中去(Linux并没有使用TSS来保存这些内容，而是使用了当前进程的系统堆栈)。可见，当地一个进程被创建后，此后只要在系统空间，其SS:ESP必然指向当前进程的系统堆栈，因为无论通过中断、系统调用还是异常进入系统空间，都会引起CPU依据当前进程的TRàGDT(n)àTSS获取该TSS中的SS0以及ESP0到CPU的SS:ESP中。

那为什么不继续使用用户空间堆栈呢？由于CPU进入系统空间之后，系统空间仅仅映射了3G--4G这段线性地址，因此其只能使用这段区间的某段内存作为堆栈，而用户空间堆栈位于0--3G范围内，故有必要设置新的堆栈，进程的系统堆栈因此诞生。

1.6 为什么通过esp&(~8191)就能得到当前进程的task_strcut？

在创建进程时， linu内核分配进程任务结构空间时，是以8KB(2个页面空间，即2^1*4KB，linux对物理内存空间和虚拟内存空间管理时，均规定其页面单位的尺寸为4KB)为单位来分配的，所以内存应用地址是8KB(2^13)的整数倍，即指针地址的低13位全为0，分配内存返回地址应该是指向struct task_struct结构，而esp&(~8191)必然指向esp其下第一个和8192对其的地址，也就是struct task_struct结构第一个字节。至于为何采用代码中的做法而不是直接将此指针保存在全局变量中以供应用，内核是从其自身的效率方面来考虑的。(One drawback to the shared-storage approach is that, for efficiency reasons, the kernel stores the 8-KB memory area in two consecutive page frames with the first page frame aligned to a multiple of 2^13. 1.7 关于AT&T ASM的内存访问

关于AT&T MASM内存访问：其标识内存的方法section:displacement(base,index,scale)表示访问内存section:[base+index*scale+displacement]。其中，base和index是任意的32-bit base和index寄存器。scale可以取值1，2，4，8。如果不指定scale值，则默认值为1。section可以指定任意的段寄存器作为段前缀，默认的段寄存器在不同的情况下不一样。如果在指令中指定了默认的段前缀，则编译器在目标代码中不会产生此段前缀代码。下面举一些例子：

1)        -4(%ebp)：base=%ebp，displacement=-4，section没有指定，由于base＝%ebp，所以默认的section=%ss，index,scale没有指定，则index为0。

2)        foo(,%eax,4)：index=%eax，scale=4，displacement=foo。其它域没有指定。这里默认的section=%ds。也可以写成foo(%eax,4)。

3)        foo(,1)：这个表达式引用的是指针foo指向的地址所存放的值。注意这个表达式中没有base和index，并且只有一个逗号，这是一种异常语法，但却合法。这里index=0，sacal=1，displacement = foo。

4)        %gs:foo：这个表达式引用的是放置于%gs段里变量foo的值。

5)        如果call和jump操作在操作数前指定前缀“*”，则表示是一个绝对地址调用/跳转，也就是说jmp/call指令指定的是一个绝对地址。如果没有指定"*"，则操作数是一个相对地址。

6)        任何指令如果其操作数是一个内存操作，则指令必须指定它的操作尺寸(byte,word,long），也就是说必须带有指令后缀(b,w,l)。

7)        需要注意的时CPU读取内存值时是向上的，而堆栈始终向下的。这样虽然esp指向栈底，但是mov esp,eax，则eax里面的是esp上面的那个内存中存的值，而不是esp下面那个内存中的值。
