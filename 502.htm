1 fork、clone和vfork

frank_seng(frank_seng@163.com)

1.1 fork和vfork区分

fork：pid_t fork(void)

vfork：pid_t vfork(void);

 

fork的实现其实就是直接调用sys_fork创建子进程，vfork的实现其实就是直接调用sys_vfork。而sys_fork和sys_vfork唯一的不同就是两者调用do_fork函数的参数不同而已；

 

asmlinkage int sys_fork(struct pt_regs regs)
{
        return do_fork(SIGCHLD, regs.esp, &regs, 0);
}

 

asmlinkage int sys_vfork(struct pt_regs regs)

{

    return do_fork(CLONE_VFORK | CLONE_VM | SIGCHLD, regs.esp, &regs, 0);

}

 

那么这种参数调用不同会导致什么结果呢？fork参数导致创建的子进程所有的一切完全复制父进程，vfork参数导致创建时，不会拷贝父进程的页表到子进程，和子进程共用用户空间栈，通常用子进程将会立刻调用execve()的情况。还有，vfork时，父进程将会挂起，直到子进程调用execve或者exit。在执行execve前，子进程共享父进程的所有用户空间内存，包括用户空间栈。信号处理是继承的而不是共享的。这里CLONE_VFORK | CLONE_VM表示父子进程共用共同的虚存空间，而且当子进程释放其虚存区间时要唤醒父进程。

1.2 fork和clone区分

fork：pid_t fork(void)

clone：int __clone(int(*fn)(void *arg), void *child_stack, int flags, void *arg)

 

而clone的实现就有些讲究了，其不但需要直接调用sys_clone创建子进程，而且在该系统调用返回后会执行函数fn，fn执行完毕后子进程就退出了。sys_clone和sys_fork唯一的不同就是两者调用do_fork函数的参数不同，以及clone需要对子进程用户空间堆栈进行特殊设置。

 

asmlinkage int sys_fork(struct pt_regs regs)
{
        return do_fork(SIGCHLD, regs.esp, &regs, 0);
}

 

asmlinkage int sys_clone(struct pt_regs regs)

{

    unsigned long clone_flags;

    unsigned long newsp;

 

    clone_flags = regs.ebx;

    newsp = regs.ecx;

    if (!newsp)

        newsp = regs.esp;  //如果child_stack=null，则使用父进程的栈

    return do_fork(clone_flags, newsp, &regs, 0);

}

 

那么这种参数调用不同会导致什么结果呢？fork参数导致创建的子进程所有的一切完全复制父进程，clone允许子进程共享父进程的执行上下文：如内存空间、文件描述符表、信号处理表。clone的主要用途就是实现线程：在一个程序实现多线程控制，从而实现在共享内存空间并发的运行。而且，clone在从系统调用返回后还会执行函数fn，fn执行完毕后子进程就退出了。

1.3 clone函数参数说明

²  fn & arg: 当子进程被创建后，其将会执行fn(arg)，void* arg参数会被穿回给fn函数。当fn函数返回时，子进程也就结束了，fn返回的整型值也就是子进程的exit code。子进程也可以在收到一个致命的信号后，调用exit(1)立刻返回。

²  child_stack: 由于子进程和父进程可能是使用共享内存，此时子进程不应该和父进程使用同样的栈，因此父进程应该为子进程分配一块内存，并将该内存的最高地址child_stack赋给clone函数作为子进程的栈。

²  flags: flags的低字节包含了当子进程死掉时，应该发送给父进程的信号的序号。其他的则按位表示了几个常量，用来指明子进程和父进程之间共享的内容：

 

CLONE_VM

如果该bit位被设置，那么子父进程运行在同样的内存空间中，父进程或子进程对内存的写，在另一个进程中都是可见的。同时，父进程或子进程的任何map或unmap都会影响到另一个进程。

如果该bit位没有被设置，则表示子进程运行在自己独立的内存空间拷贝中，和fork就类似了，一个进程对内存的读写或者映射都不会影响到另一个进程。

CLONE_FS

如果该比特位被设置，父子进程共享同样的文件系统信息。包括文件系统的根、当前工作目录以及umask。其中一个进程调用chroot，chdir，umask都会影响到另一个进程。

如果没有被设置，则子进程会复制一份父进程文件系统信息，相互之间不会产生影响。

CLONE_FILES

如果该标志位被设置，父子进程共享同样的文件描述附表。在子进程和父进程中，文件描述符总是指向同样的文件。一个进程创建的文件描述符在另一个进程中也同样有效。如果一个关闭了文件描述符，则也会影响到另一个。

如果没被设置，则子进程会复制父进程的文件描述符，之后互相不影响。

CLONE_SIGHAND

如果该标志被设置，父子进程共享同样的信号处理表。如果一个进程调用sigaction改变一个信号的行为，那么在另一个进程中该信号的行为也会相应的改变。However, the parent and child processes still have distinct signal masks and sets of pending signals.  So,  one  of  them may block or unblock some signals using sigprocmask(2) without affecting the other process.

如果没被设置，则子进程会复制父进程的信号处理表，之后互相不影响。

CLONE_PID

如果该标志被设置，创建子进程时，其ID将会和父进程的一样。

如果没有被设置，则不会一样。

 

1.4 重要注意事项

无论是fork、clone、vfork还是kernel_thread，不管选择哪种方式创建的子进程/线程都有自己独立的系统栈，而不会共享父进程的系统栈。

2 clone函数的实现

首先我们来看一段clone使用范例程序：

 

#include <sched.h>

int variable;

int do_something() {

   variable = 42;

   printf("cada\n");

   _exit(0);

}

int main(int argc, char *argv[]) {

   void *child_stack;

   char tempch;

   variable = 9;

   child_stack = (void *) malloc(400);

   printf("The variable was %d\n", variable);

   //注意这里要将参数2设置成分配内存的最高地址，因为clone要求传递的是栈顶

   clone(do_something, child_stack+400, CLONE_VM|CLONE_FS|CLONE_FILES, 0);

   sleep(1);

   printf("The variable is now %d\n", variable);

   exit(1);

}

 

clone出一个线程之后，线程将会执行函数do_something，并且使用child_stack+400处的位置作为esp栈顶。

...... 

int copy_thread(int nr, unsigned long clone_flags, unsigned long esp,

    unsigned long unused, struct task_struct * p, struct pt_regs * regs)

{

    struct pt_regs * childregs;

    // 将childregs指向子进程系统栈的栈顶

    childregs = ((struct pt_regs *) (THREAD_SIZE + (unsigned long) p)) - 1;

    // 将父进程的pt_regs复制到子进程系统栈的pt_regs中

    struct_cpy(childregs, regs);

 

    // 修改子进程的pt_regs为0，用于返回时判断当前是子进程还是父进程

    childregs->eax = 0;

    // 将子进程的用户空间堆栈修改为我们为子进程分配的堆栈

    childregs->esp = esp;

    // 设置子进程当前的esp、系统空间堆栈栈顶以及eip

    p->thread.esp = (unsigned long) childregs;

    p->thread.esp0 = (unsigned long) (childregs+1);

    p->thread.eip = (unsigned long) ret_from_fork;

 

    savesegment(fs,p->thread.fs);

    savesegment(gs,p->thread.gs);

    unlazy_fpu(current);

    struct_cpy(&p->thread.i387, &current->thread.i387);

 

    return 0;

}

......

对于父进程则按照正常系统调用方式执行，这里不再多说。其实fork以及vfork和此执行过程类似，只是fork会父进程所有的资源，包括父进程的堆栈，都是完全复制的，从sys_fork中回来，子进程的EAX也为0，父进程的不为0(pid)。由于各自空间独立，故两个进程分别正常运行了，而对于vfork而言，其实和这里的clone几乎就一样了，唯一不同的是，其没有选择自己用户控件栈的权利，其只能使用父进程的用户栈空间，这样很容易因为父进程和子进程同时使用同一个栈而导致出问题，因此在子进程结束或者调用exec之前，父进程是被堵塞住的，直到子进程结束或者调用了exec而结束，父进程才可以得以继续运行，这时候，父进程的堆栈只有一个拥有者，那就是父进程自己了。
